N1: LITEPAPER
Abstract
We present a novel layer 1 blockchain with decoupled settlement and execution layers. The settlement layer is horizontally scalable, and specially implemented to enable the execution layer to provide per-app vertical scaling. Scalability is achieved through removing the need for consensus for key primitives such as data availability and proof verification, using (unordered, asynchronous) set replication instead of (ordered, partially synchronous) log replication. Overall transaction bandwidth increases with the number of validators without compromising security. Each application is a specialized virtual machine in the execution layer operating independently and asynchronously, and communicating through message-passing. Applications can choose a variety of security mechanisms with specific security guarantees, such as succinct validity proofs, trusted execution environment attestations, and consensus. We believe that this decoupled architecture will enable a new paradigm of apps that could not exist elsewhere.
1. INTRODUCTION
N1 is a custom-built, minimal layer 1 blockchain designed to support clusters of decoupled, high-performance, application-specific virtual machines. It addresses the blockchain trilemma by decoupling consensus from execution to support horizontal scalability of the settlement layer, all without compromising security. This decomposition enables a clear separation of concerns, allowing for independent optimization and innovation across the execution and settlement layers. The execution layer facilitates state machines with arbitrary transition rules over opaque state, while the settlement layer provides essential functionality for registering and operating execution environments.
Our first key contribution is the settlement layer providing security to offchain applications. The key observation is that several primitives required for decentralized applications, such as data availability and transition verification, do not require consensus. Instead, asynchronous broadcast over a quorom is sufficient for security. We build on this to enable horizontal scalability of data availability and proof verification, limiting consensus to operations that require it. Not only does security scale with the number of validators, but so does total available bandwidth and compute.
Our second key contribution builds on the first, defining a novel execution layer. By decoupling execution from consensus and enabling parallel data availability, we enable a novel design for execution where all apps operate asynchronously. There is no notion of state contention nor locking, as apps interact through messages just as they do on the internet. Using this, we also present a crossapplication communication scheme that does not require light clients for message validity, enabling millisecond latencies and high throughput.
This work derives from our experience building decentralized exchanges as Layer 2 (L2) protocols on Ethereum, exceeding 100K transactions per second with eventual finality. To achieve this, we leveraged two key primitives:
- High-throughput data availability services to remove the transaction bandwidth bottleneck.

- Optimistic settlement with non-interactive fault-resolution [1] to remove the latency and compute bottleneck. To do so, we pioneered the use of succinct- validity fault-proofs.
Despite this, limitations from the Ethereum settlement layer still leaked to the Layer 2. Settlement latency significantly impacts L2 finalization time, and arbitrarily long transaction backlogs are unsustainable. This was especially pronounced during periods of high gas fees. Even 1 MiB transaction blocks to data availability still result in frequent settlement updates on every Ethereum block. Both bandwidth and latency constraints stem from consensus, finality, and noisy neighbour problems inherent to the Ethereum settlement layer. These issues are exacerbated as we scale the number of apps operating in a single trust zone. Apps within a single trust zone share common sequencing semantics to facilitate fast and efficient cross-chain communication. The more apps we pack into a trust zone, the greater the benefit of cross-app composition and shared liquidity.
We also observed high settlement and data-availability fees while operating our flagship high-frequency CLOB dex - NORD. Additionally, as each application has a single writer, we can leverage this to enforce ordering, all without consensus. Thus, most operations never hit consensus.

2.1. Settlement Layer.
The settlement layer provides the minimal feature set required for hosting a simple execution environment. Before explaining how users leverage N1, first, we define a glossary of terms used throughout the paper.
set replication The network provides a set replication service, where clients can write a value $v$ to an account and later perform a read to get a set including $v$. Successive reads return a superset of the previous reads. If the account is controlled by a single writer, this essentially functions as a log as the client can use the nonce to maintain order. To write a value, a client broadcasts the value to all validators and collects the signatures to assemble a certificate of strong replication, using its own network resources. The proof is then broadcast again to the network, where honest validators ensure its inclusion. This sequence only assumes asynchrony. keys, and any information regarding messages to and from other ledgers for bridging. Unconditional tick serves as a random beacon that is used by other operations in the protocol, such as random leader election and data availability challenges.
To actually store the values submitted by clients, the validators provide a simple, hybrid storage model. Each account gets namespaced access to the key-value store and can expose arbitrary endpoints to mutate or query the account state. Each account chooses the consistency flavor for its key-value store. Ordered consistency is provided for accounts needing log-based replication, and set consistency is provided for accounts that only require set inclusion for their state.
We claim that this is sufficient to construct a decentralized application with security backed by the layer 1 network. As an illustrative example, consider running the Ethereum Virtual Machine (EVM) on N1 verified using succinct validity proofs. To provide security using the settlement layer, the operator will need to
1. store transaction data with censorship resistance
2. verify the transition proof using the transaction data and prior state
3. bridge assets to and from other ledgers
Let's break down how each of these requirements is fulfilled. The operator maintains an off-chain instance of the EVM alongside its ledger. Every few blocks, the operator batches this sequence of blocks into a single data availability blob and performs the set write. The validators in the network provide data availability commitment $\mathcal{C}$ for the blob and ensure that the data remains available for the data availability period. The operator also computes a succinct validity proof $\pi$ for the transition and performs a set write, where in this case the validators also verify the proof. Any client using the EVM instance is able to query the network for and verify both $\mathcal{C}$ and $\pi$.
To bridge assets to and from other ledgers, the N1 ledger maintains an inbox and outbox of requests to each chain using the inter-blockchain communication (IBC) [6] protocol. To bridge assets in, the operator can include the light client proof of the deposit and mint the asset in the EVM instance, and to bridge assets out, the operator can use its own transition proof and push a message to the outbox. This mechanism can also be used for forced inclusion for censorship resistance, where the operator is forced to consume new messages from the inbox after a given number of layer 1 blocks and is dis-incentivized from censoring through slashing from the validators. Thus, while set replication provides safety, connecting it to the log via bridging provides liveness.
Thus, we've satisfied all the requirements. Notice that none of these operations other than bridging require consensus. This can be interpreted as the operator acting as the leader of its own ledger, but additionally with the validator set, preventing the leader from being Byzantine. Notice also that we've made few assumptions regarding the shape of the operator. The operator could be centralized for minimal latency, but could also be itself a consensus network for additional censorship-resistance guarantees. Additionally, since data availability is easily sharded, the vast majority of the data submitted to the network can horizontally scale without compromising security. We now build on this idea to show just how far we can scale the settlement layer.

2.2. Execution Layer.
The settlement layer can in practice run a multitude of execution layers, all interoperable and optimized for their specific use cases. N1 provides a canonical execution layer for applications which maximally exploits the novel properties of the settlement layer and is developed in tandem with it. The execution layer is decoupled from consensus, instead leveraging proofs to derive economic security from the validators.
Here, we present our implementation of an execution layer designed for highperformance applications. The key innovation is that all applications operate completely asynchronously, without state contention or locks. There are certain classes of applications that require real-time responses, such as decentralized exchanges competing with traditional central-limit order-books, decentralized physical infrastructure networks, and gaming. Although the design is meant to eventually leverage succinct validity proofs, we additionally describe an interim optimistic architecture and how to detect and resolve faults in this asynchronous system.
The network consists of nodes we call processes. Between two process $p$ and $q$, there may be a directed channel $c$ which acts as a FIFO queue. We assume that channels are unbounded and error-free. Each process $p$ is equipped with a state transition function $\gamma(S, a)$, where $S$ is the current state and $a$ is an action that transitions the state to a new state, and produces messages to be routed to other processes. For example, in the case of Ethereum, an action could be a block of transactions and $\gamma$ would execution this block on an EVM instance on the current world state. Messages are all delivered asynchronously, i.e. they arrive within finite but unbounded real time.
The execution layer can be thought of as a massive state machine, where the set of actions consists of all actions for each process in the system and some linearization of those actions. The problem then becomes how to create a globally consistent sequence of state snapshots. We now describe the construction.
First, each process $p$ independently acts as an application that settles to N1 just as in our example of the EVM in the previous section. Thus, each process' operator regularly batches transactions and performs writes to the settlement layer. The way we connect the processes is by introducing a new process $r$ which we call the root. The root is the process which is responsible for coordinating the collective snapshotting [7] of the entire network. We use a novel variation of the classic asynchronous snapshot to coordinate all process to take a snapshot of their state and all in-flight requests. Second, we define a notion of channel congruity, i.e. that in-flight channels are valid and are eventually delivered, and that each message is delivered exactly once or timed out.
We define two proof types:
- state transition proofs, which prove that a process $p$ is abiding by its state transition function $\gamma$. This is similar to the proofs used in the EVM example.
- channel congruity proofs, which prove that the channels are behaving reliably, i.e. are delivering messages exactly once.

--- PAGE 5 ---

Notice that for any transition of the globally networked state, it's sufficient to either have a state transition proof of an invalid transition or a channel incongruity proof to disprove the whole transition. This follows from the contrapositive of the definition of a valid state transition proof for the full asynchronous system. Thus, until succinct validity proofs become efficient enough, it's possible to use this scheme to instead perform optimistic state transitions which nonetheless derive economic security from the settlement layer.
3. CONCLUSION
By decoupling the settlement and execution layers to focus on removing consensus from the hot path, N1 provides performance and facilities not yet possible. The settlement layer is specifically design to scale to the current needs of decentralized applications, which our canonical execution layer maximally exploits. The scalability is possible thanks to providing an additional primitive, set replication as opposed to only log replication, which is sufficient for deriving safety and liveness from the validators. The execution layer is, unlike existing solutions, completely asynchronous. In other words, all applications run separately and without contention, instead communicating through messages. Since applications share the same trust zone, messages do not require mutual authentication, thus enabling immediate finality and in turn enabling new classes of applications that can exploit the real-time nature of communication.
